# i18n 辛口レビュー

現状の i18n は「最低限動く」段階で、運用と品質の観点ではかなり脆いです。  
このまま機能追加を続けると、翻訳抜け・文言不整合・多言語拡張コストが急速に増えます。

## 主要な問題点

1. **翻訳キー管理が手作業依存**
   - `backend/src/i18n.ts` に全キーをベタ書きしており、キー追加・削除時の運用が属人的。
   - 文字列資産が肥大化するとレビューで差分確認しづらい。

2. **言語フォールバックが実質2択固定**
   - `normalizeLanguage` は `ja` 以外をすべて `en` に寄せる実装で、将来言語追加時に破綻しやすい。
   - 無効値の検出や監視がないため、設定ミスを見逃す。

3. **補間変数の安全性が弱い**
   - `t()` の置換は単純な `replaceAll` のみで、未使用・不足変数の検知がない。
   - テンプレートと実データの不整合が静かに本番へ流れるリスクがある。

4. **翻訳品質の検証が不足**
   - キー整合（`en` と `ja` の完全一致）を担保する専用テストがない。
   - UI/Embed 文言の回帰を機械的に検出する仕組みが不足。

5. **ドメイン文脈が文言に混在**
   - Discord 表現・抽選ドメイン用語・運用メッセージが単一辞書に混在し、責務分離が弱い。
   - 文言改修の影響範囲が読みにくく、変更に心理的コストがかかる。

## 優先改善項目（短期）

- キー整合テストを追加し、欠落・余剰キーを CI で即検知する。
- `t()` に未解決プレースホルダ検知を導入し、開発時に失敗させる。
- 言語コードのバリデーションを明示化し、未知値はログで可観測化する。

## 優先改善項目（中期）

- 辞書を機能単位（commands / embed / interaction など）に分割する。
- 翻訳データを TS オブジェクト直書きから、検証可能な構造（スキーマ付き）へ移行する。
- 将来言語追加を前提に、フォールバック戦略（default/region variant）を設計する。

## 総評

現状は「機能実装に翻訳が後追いで貼り付いている」状態です。  
今のうちに i18n の検証・分割・可観測性を入れないと、機能追加のたびに翻訳負債が増え続けます。
